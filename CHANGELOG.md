# Changelog

All notable changes to Phirewall are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## Unreleased - 0.7.0

### Changed

- **BREAKING (behavioural): `PortableConfig::keyIp()` (and the portable `{type:'ip'}` key) now keys on the resolved client IP of the `Config` it is materialized under, not the raw `REMOTE_ADDR` it always used before.** It materializes as a keyless rule, so it late-binds to the evaluating `Config`'s IP resolver (else `REMOTE_ADDR`) at request time - aligning the portable key with keyless counter rules, `PortableConfig::filterIp()`, and every `ClientIpResolverAware` matcher (the 0.6.0 matcher change, #99). A `Config` with no IP resolver set is unaffected (resolved client IP falls back to `REMOTE_ADDR`), so every shipped preset and proxy-less deployment behaves identically. The wire format is unchanged (`keyIp()` still serializes to `{type:'ip'}`) and `typ` stays `phirewall.config.v1`, so existing serialized and signed configs load and round-trip unchanged - only their evaluation keys on the real client IP when a resolver is configured.

### Deprecated

- **`KeyExtractors::clientIp(TrustedProxyResolver)`** - the resolved client IP is now the default discriminator for every rule via the `Config` IP resolver, so a dedicated key extractor is no longer needed. Configure proxy trust once with `$config->setIpResolver($trustedProxyResolver->resolve(...))` and omit the rule's key (or use `PortableConfig::keyIp()`); both resolve the client IP.
- **`KeyExtractors::ip()`** - the name is ambiguous (it reads like "the client IP" but returns the raw `REMOTE_ADDR` peer, which behind a proxy is the proxy itself) and it bypasses the `Config` IP resolver. To key on the client IP, omit the rule's key (or use `PortableConfig::keyIp()`) so it resolves through the resolver. For the raw connecting peer, read `$request->getServerParams()['REMOTE_ADDR']` directly.

## 0.6.0 - 2026-06-17

### Changed

- **IP-aware matchers now resolve the client IP through the evaluating `Config` when no explicit resolver was set.** `IpMatcher`, `TrustedBotMatcher`, `FileIpBlocklistMatcher`, and `SnapshotBlocklistMatcher` implement the new `Flowd\Phirewall\Matchers\ClientIpResolverAware` interface: constructed without an explicit `$ipResolver`, they late-bind to the resolver of the `Config` they run under (via `Config::clientIpResolver()`), exactly like keyless counter rules resolve their key. This applies wherever such a matcher is evaluated - as a safelist or blocklist rule, as a fail2ban or track filter, and as a throttle scope filter. This removes the need to pass `ipResolver: $config->getIpResolver()` when wiring a matcher by hand, and makes IP rules composition-correct - a rule carried over from an earlier layer now honors a resolver set by a later layer, lifting the previous "composing the IP resolver does not rewrite IP-aware matchers" limitation for rules left without an explicit resolver. An explicit resolver still wins and is never overridden. **BREAKING (behavioural):** a matcher built with no explicit resolver inside a `Config` that has an IP resolver set now uses that resolver instead of the `REMOTE_ADDR` it captured at construction; pass an explicit resolver to opt out. The `SafelistSection`/`BlocklistSection` no longer take a `Config` back-reference (their `ip()`/`fileIp()`/`fromBackend()` helpers pass the explicit resolver straight through).
- **Config composition is unified behind a single method, `Config::with(ConfigLayer ...$layers): self`.** `Config::combine()` and `Config::mergedWith()` have been removed and `Config::compose()` is now private; a new `Flowd\Phirewall\ConfigLayer` interface - implemented by both `Config` and `PortableConfig` - lets live configs, portable rule data and presets compose uniformly through one call, so the caller no longer picks a method by layer type. Layers apply left to right (later wins by rule name); a `PortableConfig` layer is materialized onto the base Config's cache, dispatcher and clock, so transport/preset layers never carry a cache. Migrate `combine(...)` / `mergedWith(...)` calls to `with(...)`. **BREAKING.**

### Removed

- **Removed the route-scoped `apiRateLimiting()` and `loginProtection()` presets** and their constants (`API_RATE_LIMITING`, `LOGIN_PROTECTION`, `API_PATH_PREFIX`, `LOGIN_PATH_PREFIX`, `LOGIN_FAILURE_RULE`). They hardcoded an application's own routes (`/api`, `/login`), which vary per app, so they did not belong in a shipped, universal preset catalogue; the remaining presets (`scannerBlocking()`, `sensitivePathBlocking()`) key only on universal attacker signals. Build API rate limiting and login brute-force protection as plain `Config` instead (see `examples/03-api-rate-limiting.php` and `examples/02-brute-force-protection.php`). **BREAKING.**
- **The OWASP CRS / ModSecurity `SecRule` engine was extracted into a companion package.** The `Flowd\Phirewall\Owasp\` namespace (`SecRuleLoader`, `CoreRuleSet`, `CoreRuleSetMatcher`, the operator and variable collectors) and the `$config->blocklists->owasp()` shortcut have been removed from core. OWASP CRS support now lives in [`flowd/phirewall-preset-owasp-crs`](https://github.com/flowd/phirewall-preset-owasp-crs), which ships the engine under `Flowd\PhirewallPresetOwaspCrs\Engine\` plus ready-made per-paranoia-level CRS presets. To migrate: `composer require flowd/phirewall-preset-owasp-crs` and either apply a preset with `$config = $config->with(Presets::blocklist(ParanoiaLevel::Level1))`, or register a rule set yourself with `$config->blocklists->addRule(new \Flowd\Phirewall\Config\Rule\BlocklistRule('owasp', new \Flowd\PhirewallPresetOwaspCrs\Engine\CoreRuleSetMatcher($ruleSet)))`. The OWASP diagnostics header (`X-Phirewall-Owasp-Rule`, `enableOwaspDiagnosticsHeader()`) stays in core and works unchanged with the companion engine. **BREAKING.**
- **Removed the `$config->safelists->trustedBots()` convenience method.** The reverse-DNS `TrustedBotMatcher` it wrapped stays in core, so wire it directly: `$config->safelists->addRule(new \Flowd\Phirewall\Config\Rule\SafelistRule('trusted-bots', new \Flowd\Phirewall\Matchers\TrustedBotMatcher(ipResolver: $config->getIpResolver(), cache: $cache)))`. Pass `ipResolver: $config->getIpResolver()` to keep the behaviour `trustedBots()` had by default (correct client IP behind a proxy); omit it to key on `REMOTE_ADDR`. Using the matcher directly also lets you rate-limit verified bots instead of safelisting them (return its key from a throttle's key closure). **BREAKING.**

## 0.5.0 - 2026-06-08

### Added

- **Presets: ready-to-use rule bundles for common scenarios**: `Flowd\Phirewall\Preset\Presets` ships four curated rulesets, each a `PortableConfig` (inspectable, serializable, signable) returned by an accessor: `Presets::apiRateLimiting()`, `loginProtection()`, `scannerBlocking()`, `sensitivePathBlocking()`, or `Presets::get($name)` by name. Materialize one or several onto your own cache with `Config::combine()`; presets are pure data and never receive a cache. Every rule is namespaced `preset.<area>.*` so a later layer that redefines it by name overrides predictably. The presets: `apiRateLimiting()` (per-client sliding-window burst + sustained throttles scoped to `/api`), `loginProtection()` (a `/login` throttle plus a brute-force fail2ban that bans an IP after 5 failures in 15 min, driven solely by `RequestContext::recordFailure(Presets::LOGIN_FAILURE_RULE)`; the rule uses a never-match filter so a spoofable marker header can never trip it), `scannerBlocking()` (known scanner/exploit User-Agents + missing standard browser headers), and `sensitivePathBlocking()` (a pattern blocklist for `/.git`, `/.env*`, `/.aws/credentials`, `/.htpasswd`, … probes). A `Presets::VERSION` catalogue version lets integrators detect ruleset updates; Phirewall hardcodes no remote endpoint and performs no network I/O, and comparing it against a release feed you trust (Packagist, a config service, a versioned URL) with `version_compare()` is the integrator's job. See `examples/31-presets.php`.
- **`PortableConfig::filterNone()`**: A never-match filter factory (the portable counterpart of `fn() => false`), symmetric to `filterAll()`. Use it for a fail2ban rule that must not be tripped by any inspectable, client-controlled request property and is driven solely by a trusted post-handler signal via `RequestContext::recordFailure()`. Backs the hardened `loginProtection()` preset.
- **`PortableConfig::throttle()` gained an optional `scope` filter**: Passing a `$scope` filter (any of the existing filter factories, e.g. `PortableConfig::filterPathPrefix('/api')`) restricts which requests a throttle counts: matching requests are throttled per the rule's key, all others pass through untouched by it. This keeps a per-client throttle from accidentally rate-limiting unrelated traffic and is the building block behind the path-scoped throttles in the new presets. The addition is backward compatible: throttles without a scope behave exactly as before, and the schema only carries `scope` when set.
- **`KeyExtractors::hashedHeader(string $name)`**: Returns a key extractor that yields the sha256 fingerprint of the named header value instead of the raw value. Preferred over `KeyExtractors::header()` whenever the header is credential-bearing (`Authorization`, `Cookie`, `X-Api-Key`, …), so the cache backend and the ban registry store the fingerprint rather than the verbatim value. The hash is unkeyed sha256, a deterministic bucket identifier, not a credential-hardening primitive: low-entropy values remain grindable from a leaked cache dump, and the fingerprint does not defeat a chosen-key probing attacker. The `header()` docblock now points readers at this helper.
- **`RequestContext::recordHit()`**: Counterpart of `recordFailure()` for allow2ban rules. Handlers can now signal handler-observable hits (e.g. an expensive operation completed, a webhook delivered a duplicate payload) and have them counted against an allow2ban threshold post-handler, mirroring the fail2ban-via-RequestContext pattern.
- **`PortableConfig` signed transport (HMAC-SHA256)**: `PortableConfig::toSignedJson()` produces a JWS-compact-style envelope (`<header>.<payload>.<signature>`) and `PortableConfig::loadSigned()` verifies it with `hash_equals()` constant-time compare. Intended for the case where the serialized config is read from storage the caller does not fully control (shared filesystems, S3, etcd, config services). The existing `fromArray()` / `toArray()` round-trip is unchanged; the `fromArray()` docblock now points operators at the signed helpers when the input source is not fully trusted. Signing keys must be at least 16 bytes; 32 random bytes is recommended.
- **Config composition / layering**: `Config::compose(self $base, self ...$overlays): self` and the fluent `Config::mergedWith(self ...$overlays): self` merge several `Config` instances into one effective `Config` without mutating any input. The motivating use case is stacking a vendor baseline + environment overlay + tenant overlay + per-deployment tweak (each often rebuilt from a `PortableConfig`). Overlays apply left to right, so **later sources win**: rules merge by name within every section (safelists, blocklists, throttles, fail2ban, allow2ban, tracks), a later rule replaces the earlier same-named rule in place, preserving base ordering and appending genuinely new rules (a union, not a concat-with-duplicates); pattern backends merge by name the same way; scalar/object options (`keyPrefix`, `failOpen`, the response-header toggles, the IP resolver, the discriminator normalizer, the response factories) follow "last explicit value wins" so a layer that left an option at its default never clobbers an earlier explicit value. Infrastructure (PSR-16 cache, PSR-14 dispatcher, clock) is inherited from the base layer. `Config::combine(PortableConfig ...$portableConfigs): self` is the PortableConfig-accepting sibling: it materializes each portable on the receiver's cache and composes them with the same precedence, so portable rule data and presets stay pure: the cache lives only on `Config`, never on the transport/preset layer. See `examples/30-config-composition.php`.
- **`PortableConfig` promoted to a first-class transport**: The portable schema now expresses most rulesets people actually write, not just a narrow subset. **New rule types:** `allow2ban()` (volume cap) and pattern backends (`addPatternBackend()` / `blocklistFromBackend()` / `patternBlocklist()` with all eight `PatternKind` kinds, the building block for storing block catalogues in a database). **New filters:** `filterPathPrefix()`, `filterPathRegex()`, `filterMethodIn()`, `filterHeaderPresent()`, `filterHeaderRegex()`, plus matcher-backed `filterIp()` (CIDR-aware `IpMatcher`), `filterKnownScanners()`, and `filterSuspiciousHeaders()`. **New key extractor:** `keyHashedHeader()`. **New options:** `throttle()` gained a `sliding` flag, plus `setFailOpen()` and `enableOwaspDiagnosticsHeader()`. Materialization registers rules through the section API instead of the deprecated `addSafelist()` / `addBlocklist()` / … forwarding methods, and uses the dedicated matcher classes where a filter maps onto one. A `PortableConfig` is materialized onto a `Config` you build (with its cache) via `Config::combine()`: the schema itself never takes a cache. All additions are backward compatible: existing serialized schemas import unchanged. Trusted-bot reverse-DNS matchers, OWASP CRS rulesets, file-backed lists, and closure-driven dynamic throttle limits remain intentionally out of scope (not serializable as pure data). See `examples/29-portable-config.php`.
- **PSR-16 key validation on all cache backends**: `InMemoryCache`, `PdoCache`, `RedisCache`, and `ApcuCache` now validate every key on their PSR-16 surface (`get` / `set` / `has` / `delete` / `getMultiple` / `setMultiple` / `deleteMultiple`) and throw `Flowd\Phirewall\Store\InvalidCacheKeyException` (which implements `Psr\SimpleCache\InvalidArgumentException`) for an invalid key: an empty string, a key containing a reserved character (`{}()/\@:`) or a control/whitespace character, or, for the multi-key methods, a non-string key (these previously cast keys to string silently). Per PSR-16 there is no upper length limit; keys longer than the mandated 64-character minimum remain valid. Validation lives in a shared `KeyValidationTrait` so the rules are identical across backends. **BREAKING:** a reserved/control/whitespace key, or a non-string key passed to a multi-key method, was previously cast to string silently and now throws; sanitise any custom cache keys to the PSR-16-safe set before upgrading.
- **The `$key` argument is now optional on `throttles->add()` / `sliding()` / `multi()`, `fail2ban->add()`, `allow2ban->add()`, and `tracks->add()`**: When omitted, the rule keys on the client IP resolved by the `Config`'s IP resolver (set via `Config::setIpResolver()`, typically `KeyExtractors::clientIp($trustedProxyResolver)` behind a proxy), falling back to `KeyExtractors::ip()` (REMOTE_ADDR) when none is configured. This makes proxy-aware per-client keying the zero-config default and removes the most common source of copy-pasted `key: KeyExtractors::ip()` boilerplate. A keyless rule stores a `null` key extractor and is resolved per request against the `Config` it is evaluated under (via the new `Config::resolveKey()`), so the IP resolver may be set before or after the rule, and a composed `Config` (`Config::compose()`) applies its own merged IP resolver to default-keyed rules carried over from any layer. An empty-string discriminator is normalized to `null` (the rule is skipped). Passing an explicit `$key` behaves exactly as before, so existing code is unaffected.

### Changed

- **BREAKING `TrustedProxyResolver` default `$allowedHeaders` is now `['X-Forwarded-For']`**: Previously the default was `['X-Forwarded-For', 'Forwarded']` and the resolver returned on the first non-empty header in declared order. Operators on stacks that emit RFC 7239 `Forwarded` should pass `['Forwarded']` (or both names) explicitly so the chosen header is visible in the call site rather than inferred from a default. Migration: integrators that relied on the implicit `Forwarded` fallback need to opt in by passing the value at the construction site.
- **BREAKING `Config::setKeyPrefix()` rejects prefixes containing PSR-16 reserved characters**: The prefix is concatenated verbatim into every generated cache key, so `setKeyPrefix()` now fails fast (throwing `\InvalidArgumentException`) when the prefix contains a PSR-16 reserved character (`{}()/\@:`), a control character, or whitespace, naming the offending character at the call site instead of letting it surface later as an `InvalidCacheKeyException` on the first cache operation. A multi-segment colon prefix such as `setKeyPrefix('app:prod')` (a common Redis namespacing style) was silently accepted before and now throws. A single trailing `:` is still stripped for convenience (so `'phirewall:'` remains valid). **Migration:** rewrite colon-namespaced prefixes with `.`, e.g. `app:prod` → `app.prod`.
- **BREAKING `RequestContext::recordFailure()` second parameter is now optional and the recorded type renamed**: When `$key` is omitted, the firewall extracts the discriminator from the matching rule's `keyExtractor` against the current request, so handlers no longer need to know whether the rule keys on IP, header, or anything else. `RecordedFailure` was renamed to `RecordedSignal` and gained a `BanType $banType` field so the same recorder can route fail2ban and allow2ban signals through the same channel. `RequestContext::getRecordedFailures()` is replaced by `getRecordedSignals()`; integrators iterating recorded signals manually (rare, `Middleware::process()` does this internally) need to update the method name and adapt to the new field on the returned objects. The existing explicit-key form (`recordFailure($name, $key)`) continues to work unchanged.
- **BREAKING `isBanned()` requires an explicit `BanType`**: `Http\Firewall::isBanned()` and `BanManager::isBanned()` previously defaulted `$banType` to opposite values (`Fail2Ban` vs `Allow2Ban`), so calling the wrong one silently queried the wrong ban namespace and returned a false negative. The default is removed on both; callers must pass `BanType::Fail2Ban` or `BanType::Allow2Ban` explicitly.
- **BREAKING `SuspiciousHeadersMatcher` and `BlocklistSection::suspiciousHeaders()` use `null` for "use defaults"**: they previously signalled "use the curated default header set" with an empty array (`[]`), the opposite of the sibling `KnownScannerMatcher` / `knownScanners()` which use `?array $patterns = null`. Both now standardise on `?array = null` (null means curated defaults, a non-empty list overrides); pass `null` instead of `[]` for defaults. An explicit empty list (`[]`) no longer means defaults: it now requires no headers, so the matcher never matches (effectively disabled). `PortableConfig::filterSuspiciousHeaders([])` expresses that same intent and round-trips through `toArray()`/`fromArray()` rather than throwing.

- **BREAKING Internal cache-key separator changed from `:` to `.` (one-time counter reset on upgrade)**: To make Phirewall's own keys PSR-16-compliant (the spec reserves `:` for the cache implementation, not its callers), `CacheKeyGenerator` and `TrustedBotMatcher` now join key segments with `.` instead of `:`, and `CacheKeyGenerator::normalizeName()` no longer keeps `:` from a rule name. The visible effect: throttle counters, fail2ban/allow2ban fail/hit/ban counters, the ban registry, and the trusted-bot rDNS cache are keyed differently after upgrading, so these **ephemeral, TTL-bound entries reset once**: in-flight throttle windows restart and, because the ban registry re-keys, **active fail2ban/allow2ban bans are forgotten on the first deploy** (a currently-banned client is no longer blocked until it re-triggers the rule). This is a **one-time enforcement gap that self-heals**: every affected entry is short-lived and is rebuilt on the next matching request, and there is no API change. `RedisCache`'s namespace prefix (default `Phirewall:`) is unaffected: it is the backend's own keyspace, applied after the public key is validated.
- **`BanManager` registry has a TTL and prunes expired entries on save**: Previously the per-rule ban registry (the audit-view list backing `listBans()` / `listRulesWithBans()`) was written without a TTL and grew monotonically across ban churn. Each save now applies a TTL equal to the longest-surviving entry's remaining lifetime, and `saveRegistry()` prunes already-expired entries before writing so the registry tracks live bans only. The registry is best-effort under concurrent `ban()` calls for the same rule (the primary ban cache key set by `ban()` is the source of truth and is not affected).
- **`FileIpBlocklistMatcher` and `InfrastructureBanListener` docblocks call out the REMOTE_ADDR default**: Both constructors fall back to `KeyExtractors::ip()` when no `$ipResolver` / `$requestToIp` is supplied. That extractor reads `REMOTE_ADDR` verbatim, which behind a CDN or load balancer is the proxy's address rather than the client's. The constructor docblocks now point integrators at `KeyExtractors::clientIp(new TrustedProxyResolver([...]))` for the proxy case so the blocklist match (and any infrastructure ban that follows) targets the originating client.
- **Quick-Start fail2ban example and `examples/02-brute-force-protection.php` use `RequestContext`**: Both previously matched on a request header (`X-Login-Failed: 1`) as the failure signal, but the firewall evaluates fail2ban filters before the handler runs, so a real attacker never sends that header and no ban ever fires. Example 02 only appeared to work because the test harness pre-injected the marker into simulated requests. The recommended `RequestContext::recordFailure()` pattern is now the one shown: the handler retrieves the context attribute, calls `$context->recordFailure('login-failures', $remoteAddr)` on a failed login, and Middleware applies the signal after the response is built.
- **Pattern blocklist read hot path reuses a cached snapshot**: `FilePatternBackend::consume()` reuses its parsed snapshot while the file's modification time is unchanged (rebuilding only when the file changes or the earliest pending entry expiry passes) instead of re-reading and re-parsing on every request, and `SnapshotBlocklistMatcher` resolves header-targeted patterns via the PSR-7 case-insensitive header accessor rather than building a full lowercased header map per request. In-process updates (`append`/`pruneExpired`) invalidate the cache immediately; like the file IP blocklist's reload gate, an external rewrite is reflected on the next modification-time change.
- **OWASP CRS evaluation collects each request variable once per request, with a configurable per-variable value cap that fails closed**: `CoreRuleSet` now derives each request variable (`ARGS`, headers, …) a single time per request through a shared `RequestVariableValues` memo instead of re-deriving it for every rule. The number of values evaluated per variable is bounded (configurable via the new `$maxValuesPerCrsVariable` argument, a positive integer, on `CoreRuleSet` and `SecRuleLoader::from*()`), so a client cannot amplify per-request rule-evaluation cost by sending more values (collection still scales with the request up to the runtime's own input limits; it is the rule evaluation that is bounded). The cap counts collected values and defaults to **twice** PHP's `max_input_vars`: variables such as `ARGS` emit both a value and a name per parameter (~2 values per input var), so doubling the directive sizes the budget to the parameter count the runtime accepts: any request PHP can fully parse stays fully inspected. The cap is per variable, not aggregate, padding one variable cannot short-circuit evaluation of a rule targeting another, and when a variable a deny rule inspects is truncated, the rule **fails closed** (the un-inspectable oversized request is treated as a match) rather than letting a payload padded past the cap slip through unevaluated.
- **Fail2ban and allow2ban evaluators batch ban-key existence checks**: the `Fail2BanEvaluator` and `Allow2BanEvaluator` now collect all candidate ban keys for the request and issue a single `getMultiple()` (one `MGET` on Redis) instead of a per-rule `has()`, then only the rules whose filter matches increment their counter. On the common unbanned path this collapses K per-rule round-trips into one. Behaviour is preserved; allow2ban still iterates every rule so all counters advance, capturing the first block via a small `Allow2BanDecision` value object.
- **Shared PSR-16 bulk cache operations, with per-key failure reporting**: the trivial bulk methods (`getMultiple`/`setMultiple`/`deleteMultiple`) shared by `InMemoryCache` and `ApcuCache` now live in a single `BulkCacheOperationsTrait`. `PdoCache` also uses the trait for `setMultiple()` while keeping batched `getMultiple()` / `deleteMultiple()` overrides, and `RedisCache` serves `getMultiple()` / `deleteMultiple()` with native `MGET` / `DEL` while its `setMultiple()` loops over per-key `set()` (`MSET` carries no per-key TTL). The shared `setMultiple()` / `deleteMultiple()` now return `false` when an underlying per-key operation fails (e.g. `apcu_store()` returning `false`) instead of always returning `true`, matching PSR-16 per-key failure semantics (`InMemoryCache` is unaffected in practice because its `set()` / `delete()` always succeed). `InMemoryCache` also routes its clock fallback through a single `now()` helper, and `RedisCache`'s value-encoding contract is now documented.

### Security

- **`RedisCache::increment()` re-throws on Redis errors instead of returning `0`**: Previously any `\Throwable` from the underlying Predis client (connection refused, AUTH failure, Lua script error, network blip) was swallowed and `increment()` returned `0`, which every throttle / fail2ban / allow2ban rule then interpreted as "no hits this window". The method now re-throws the original `\Throwable` after emitting an `E_USER_WARNING` for diagnostic visibility, so `Middleware::process()` applies the configured `Config::failOpen` policy uniformly for Redis errors. Integrators relying on the prior fail-open behaviour can keep `Config::failOpen` at its current default of `true`; those wanting the failure to surface as a 5xx can set it to `false`.
- **Observability examples extract a curated subset instead of dumping the event**: `examples/09-observability-monolog.php` previously called `get_object_vars($event)` and passed the result to Monolog, which serialised the embedded `ServerRequest` (every header + parsed body) into the log sink. The example now uses an explicit `summarize()` helper that extracts only the rule name, method, path, remote address, and a short sha256-prefix fingerprint of the discriminator key. `examples/10-observability-opentelemetry.php` switched from `phirewall.key` to `phirewall.key_fingerprint` for the same reason.
- **`TrustedProxyResolver` keeps the rightmost `maxChainEntries` of an oversized chain, not the leftmost**: When `X-Forwarded-For` (or `Forwarded`) carried more entries than `maxChainEntries` (default 50), the parser truncated from the left and returned early, dropping the entries closest to the receiver. Because those rightmost entries are the ones added by proxies along the request path, the truncation could discard the authoritative tail in favour of arbitrary leading content. The parser now slices to the last N entries before walking the chain right-to-left.
- **BREAKING `@pmFromFile` confines resolved paths to the configured context folder**: When `PhraseMatchFromFileEvaluator` is constructed with a `$contextFolder`, the operand is now rejected if it is an absolute path, and the resolved path (via `realpath()`) is verified to remain under `realpath($contextFolder)`. Catches symlinks pointing outside the rules directory in addition to the existing `..` traversal check. `SecRuleLoader::fromFile()` now derives the loaded file's own directory as the context folder (mirroring `fromFiles()`/`fromDirectory()`), so single-file loads are confined too. When `$contextFolder` is `null` (only `SecRuleLoader::fromString()` and direct in-process construction), the prior behaviour is unchanged. **Migration:** rules that referenced a phrase file by an absolute path or via a `..` escape now throw on load; move the referenced file inside the rule file's directory and reference it relative to that folder.
- **`ApacheHtaccessAdapter` serialises concurrent writers via `flock()`**: `blockMany()` / `unblockMany()` now acquire an exclusive advisory lock on a sidecar `${htaccessPath}.lock` file for the read-modify-write cycle that culminates in the atomic rename. Two processes invoking `blockIp()` simultaneously will now serialize on the lock instead of clobbering each other's update. The lock file is created on first use and never renamed, so it remains a stable lock target across writes.
- **`FilePatternBackend` and `FileIpBlocklistStore` write via temp file + atomic rename, serialized by a sidecar lock**: Both stores previously wrote the new contents by `ftruncate(0) + fwrite` on the live file handle. A PHP fatal between the truncate and the fwrite would leave the file empty (every entry the operator had recorded silently lost). The full-rewrite path now writes to `${filePath}.tmp.<random>` and `rename()`s onto the target so the live file is either the old content or the new content, never an empty in-between state. Writers now serialize on a dedicated, never-renamed `${filePath}.lock` sidecar file rather than on the live file: because `rename()` swaps the live file's inode, a lock held on the live file would be stranded on the orphaned inode and a second writer already blocked on it would resume against stale content and clobber the just-completed update. The temp file is also `chmod()`ed to match the live file's mode (owner-only `0600` when it does not yet exist) so the rename never widens a restrictive permission set. `FileIpBlocklistStore`'s append-only path is unchanged because a trailing partial line is tolerated by the parser and reconciled on the next rewrite.
- **BREAKING Pattern and blocklist auto-created directories use mode `0700` instead of `0777`**: `FilePatternBackend::ensureDirectory()` and `FileIpBlocklistStore::ensureDirectory()` previously created missing parent directories world-readable / writable. The default is now `0700` so the contents are restricted to the owning user on shared hosts. **Migration:** if a different OS user reads the store files than the one that writes them (for example a web server user reading files written by a deploy user), run both as the same user or pre-create the directory with the wider permissions you need (auto-create only fires when the path does not yet exist).
- **BREAKING IP matching collapses IPv4-mapped IPv6 peers to their embedded IPv4 form before lookup**: Rules written as `1.2.3.4` (or CIDR `10.0.0.0/24`) now match peers that PHP presents as `::ffff:1.2.3.4` (or `::ffff:10.0.0.50`). Dual-stack PHP-FPM pools listening on AF_INET6 frequently surface IPv4 clients in the mapped form, and operators previously had to either duplicate every rule in IPv6 notation or reconfigure the SAPI to deliver plain IPv4. The collapse is shared via `CidrMatcher`, so it applies uniformly to `IpMatcher` and, critically for blocklisting, to the file and snapshot blocklist matchers (`FileIpBlocklistMatcher`, `SnapshotBlocklistMatcher`), for both exact-IP and CIDR entries. Rules and matching share one canonical binary representation, so existing IPv6-only rules and pure-IPv4-stack deployments are unaffected. **Migration:** equivalent IPv6 spellings now resolve to one identity, so re-verify any IPv6 allow/deny lists and expect counter buckets keyed on a canonicalised client IP.
- **`TrustedProxyResolver` canonicalises IPv6 entries in both `$trustedProxies` and the inbound chain**: Bare-IP trusted-proxy entries are pre-converted to `inet_pton` binary at construction time and the trust check looks the peer up in that set, so alt-form IPv6 (`2001:0db8:0:0:0:0:0:1` vs `2001:db8::1`, mixed case) and the IPv4-mapped IPv6 dual-stack presentation all resolve to the same identity. `normalizeIp()` additionally rewrites the IP to the canonical `inet_ntop` form so the resolved client IP that flows into throttle / fail2ban / event keys is consistent across requests regardless of how the proxy spelled it. CIDR entries continue to use the existing `CidrMatcher` path.
- **BREAKING `PortableConfig` rejects `header_equals` filters in safelists**: A safelist matching on a static header value is effectively a bypass token: any request presenting the configured header value skips every downstream rule, and the value sits in the rules file in plaintext. `PortableConfig::safelist()` and `PortableConfig::fromArray()` now throw `InvalidArgumentException` when a safelist filter uses `header_equals`. The filter type is still available for blocklist / fail2ban / track rules where it restricts rather than relaxes; the `header` key extractor on counter-based rules is unchanged. **Migration:** a serialized config carrying a `header_equals` safelist now fails to load; switch that safelist to an IP filter (`filterIp()`) or an authenticated-principal check, or move the rule to a blocklist/fail2ban/track section.
- **`header_equals` value comparison uses `hash_equals()`**: The compiled `header_equals` closure now compares the incoming header value to the configured value with `hash_equals()` for constant-time behaviour, eliminating the trivial timing oracle on `===` for short header values.

### Removed

- **Removed the unused `PresetUpdateChecker` seam**: `PresetUpdateChecker` and `NullPresetUpdateChecker` had no library consumer (nothing accepted or called them). They are removed; integrators compare `Presets::VERSION` against their own release feed with `version_compare()` (see the `Presets` docblock and `examples/31-presets.php`).
- **BREAKING Removed the deprecated `Config` forwarding methods (`DeprecatedConfigMethods`)**: the pre-sections convenience methods on `Config` are removed. Use the section API instead: `safelist()` → `safelists->add()`, `blocklist()` → `blocklists->add()`, `throttle()` → `throttles->add()`, `fail2ban()` → `fail2ban->add()`, `track()` → `tracks->add()`, `owaspBlocklist()` → `blocklists->owasp()`, the `addSafelist()` / `addBlocklist()` / `addThrottle()` / `addFail2Ban()` / `addTrack()` typed registrations → the matching `…->addRule()`, the `getSafelistRules()` / `getBlocklistRules()` / `getThrottleRules()` / `getFail2BanRules()` / `getTrackRules()` / `getPatternBackends()` getters → the section `…->rules()` / `blocklists->patternBackends()`, and the response-factory helpers (`blocklistedResponse()` / `throttledResponse()` / `setBlocklistedResponseFactory()` / `getBlocklistedResponseFactory()` / `setThrottledResponseFactory()` / `getThrottledResponseFactory()`) → assign or read the public `Config::$blocklistedResponseFactory` / `Config::$throttledResponseFactory` properties directly (wrapping a closure in `ClosureBlocklistedResponseFactory` / `ClosureThrottledResponseFactory`). The deprecated diagnostics forwarders are replaced by a `DiagnosticsDispatcher` wrapping `DiagnosticsCounters`.

### Fixed

- **`TrustedProxyResolver` resolves bracketed IPv6+port forms in `X-Forwarded-For` and `Forwarded`**: Entries like `[2001:db8::1]:443` (the form RFC 7239 mandates for IPv6 in `Forwarded for=`, and one some proxies emit in XFF) were silently dropped: bracket-stripping left `2001:db8::1]:443`, which failed `FILTER_VALIDATE_IP`, and the resolver fell back to `REMOTE_ADDR`. `normalizeIp()` now extracts the address from `[…](:port)?` before validating, and the `Forwarded for=` regex no longer over-captures the trailing `]`.
- **`TrustedProxyResolver` flattens all `X-Forwarded-For` / `Forwarded` header instances before the trusted-hop walk**: it gathers every received instance into one chain, so folded (RFC 7230 §3.2.2, nginx's default, and many `$_SERVER`-derived PSR-7 factories) and unfolded multi-instance forms resolve identically. Anti-spoofing comes solely from the strict right-to-left walk stopping at the first untrusted hop, not from any assumption about how header instances are separated.

## 0.4.0 - 2026-05-19

### Changed

- **BREAKING CHANGE: Ban evaluator threshold semantics unified to `>=`**: `Fail2BanEvaluator` (pre-handler path) and `Allow2BanEvaluator` previously banned only after the threshold was **exceeded**: on the **(N+1)th** matching request when `threshold = N` (they used `$count > $threshold`). They now ban as soon as the threshold is **reached**: on the **Nth** matching request (using `$count >= $threshold`), matching the post-handler fail2ban path, the diagnostics events, and classic fail2ban semantics where `maxretry = N` means "ban at the Nth attempt".

  `ThrottleEvaluator` is **not** affected: throttle still uses `$count > $limit` because "N requests per period, 429 on the (N+1)th" is the standard HTTP rate-limit contract.

  **Migration:** if you relied on the old behavior (e.g. `threshold: 5` to mean "ban on the 6th request"), subtract 1 from your `threshold` values to retain it.

  The internal `Fail2BanEvaluator::incrementAndBanIfNeeded()` method also drops its `bool $postHandler` parameter: the two paths now share the same "ban when the threshold is reached" semantic. Callers outside `Firewall` that passed this argument must remove it.

### Fixed

- **`RegexEvaluator` no longer treats literal first/last characters of `@rx` patterns as PCRE delimiters**: The previous `ensureRegexDelimiters()` heuristic checked whether a pattern's first and last characters were the same non-alphanumeric, non-bracket character and, if so, returned it unchanged on the assumption that it was already `~…~`-style delimited. ModSecurity/CRS `@rx` patterns are bare regex content by spec, so this heuristic could misfire on rules whose patterns happen to start and end with the same literal character. The clearest case is **CRS 942510** ("SQLi bypass attempt by ticks or backticks"), whose pattern is wrapped in literal backticks; under the old code those backticks were consumed as PCRE delimiters and the rule collapsed to its inner alternation, which matched almost any HTTP value. The `@rx` operator now always wraps in `~…~u` and escapes unescaped `~`. **API change:** the public static `RegexEvaluator::ensureRegexDelimiters()` has been removed; use `RegexEvaluator::wrapInTildeDelimiters()` if you need the helper directly.
- **`SecRuleParser` no longer strips literal quote characters from `@rx` arguments**: The parser previously ran `stripQuotes()` on every operator argument, which corrupted `@rx` patterns whose first and last bytes were matching `'` or `"`. The clearest case is **CRS 942511** (a variant of 942510), whose alternation is wrapped in literal apostrophes; the parser ate those apostrophes before the pattern reached `RegexEvaluator`, collapsing the rule to its inner body and matching essentially every HTTP value. `@rx` arguments are bare PCRE per ModSecurity grammar: the outer SecRule-level quoting is removed once at the line level and there is no second quoting layer to strip. The 942510 fix only addressed the evaluator-side delimiter heuristic; this completes the fix on the parser side. Non-`@rx` operators retain the existing `stripQuotes` + `unescape` behavior.

## 0.3.0 - 2026-04-08

### Added

- **Config section API**: New `$config->safelists`, `$config->blocklists`, `$config->throttles`, `$config->fail2ban`, `$config->tracks` section objects replace flat `add*()` methods. Old methods are retained but deprecated for removal in 0.5. (#1)
- **PSR-14 diagnostics**: `DiagnosticsCounters` observer with `DiagnosticsDispatcher` wrapper for event counting and forwarding. Tracks safelisted, blocklisted, throttled, fail2ban, allow2ban, and track events. (#1, #30, #33)
- **KnownScannerMatcher**: Block sqlmap, nikto, nmap, burp, and other scanner User-Agents via `$config->blocklists->knownScanners()`. (#5)
- **TrustedBotMatcher**: Safelist Googlebot, Bingbot, etc. via reverse DNS verification. (#6)
- **SuspiciousHeadersMatcher**: Block requests missing standard browser headers. (#6)
- **IpMatcher**: Safelist or block by IP/CIDR range with shared matcher utilities. (#6)
- **Allow2Ban**: Hard volume cap with auto-ban after threshold. Includes BanManager API, validation, and security hardening. (#7)
- **Sliding window throttle**: `SlidingWindowStrategy` for smoother rate limiting alongside existing fixed-window strategy. (#8)
- **Multi-window throttle**: `$config->throttles->multi()` for burst + sustained rate limiting with multiple windows. (#9)
- **Dynamic throttle limits**: Closure-based limits that can vary per request (e.g., by user role). (#9)
- **Throttle disable toggle**: Disable individual throttle rules without removing configuration. (#9)
- **Discriminator normalizer**: Normalize throttle/fail2ban keys (e.g., lowercase, trim) via callable. (#9)
- **Track with threshold**: `$config->tracks->add()` now supports optional `limit` parameter with `thresholdReached` flag for passive counting with alerting. (#10)
- **PdoCache**: SQL-backed PSR-16 cache for MySQL, PostgreSQL, and SQLite with probabilistic pruning and WAL mode support. (#11)
- **Fail-open/fail-closed**: Configurable middleware error handling: fail-open (default) continues on cache errors, fail-closed returns 503. Errors dispatched via PSR-14. (#12)
- **PSR-17 response factories**: `$config->usePsr17Responses()` and dedicated `Psr17BlocklistedResponseFactory` / `Psr17ThrottledResponseFactory` for framework-native responses. (#14)
- **RequestContext**: Post-handler fail2ban signaling via request attributes. Record failures from your login handler without pre-filtering. (#22)
- **DecisionPath enum**: Type-safe representation of all firewall decision outcomes. (#16)
- **Evaluator classes**: Dedicated per-rule-type evaluator classes (`SafelistEvaluator`, `BlocklistEvaluator`, `ThrottleEvaluator`, `Fail2BanEvaluator`, `Allow2BanEvaluator`, `TrackEvaluator`) extracted from `Firewall::decide()` for SRP. (#31)
- **PatternKind enum**: `PatternKind` converted from class with string constants to a string-backed enum. `PatternEntry` gains `key()` and `merge()` methods, deduplicating logic from pattern backends. (#32)
- **ReDoS length guard**: `RegexEvaluator` skips values exceeding 8 KiB before running regex, preventing catastrophic backtracking on crafted payloads. Reuses `RegexMatcher::MAX_SUBJECT_LENGTH`. (#34)

### Changed

- **Response headers are opt-in**: `X-Phirewall`, `X-Phirewall-Matched`, and `X-Phirewall-Safelist` headers are no longer added by default. Call `$config->enableResponseHeaders()` to enable them. `Retry-After` remains always-on for throttled responses. **Breaking change.** (#25)
- **Fail2Ban/Allow2Ban threshold semantics**: Ban now triggers after N failures (threshold reached), not on the Nth failure attempt. This is a correctness fix but changes observable behavior for edge cases. (#26)
- **RedisCache::increment()**: Now emits `E_USER_WARNING` via `trigger_error()` on Redis errors before returning 0 (fail-open preserved, including when error handlers throw). (#35)
- **Constructor validation**: All rule types (`ThrottleRule`, `Fail2BanRule`, `Allow2BanRule`, `TrackRule`, `SafelistRule`, `BlocklistRule`) now validate parameters on construction. (#18)
- **OWASP operator preprocessing**: `@rx`, `@pm`, `@pmFromFile` operators cache compiled patterns for reuse. (#20)
- **Discriminator normalizer**: Resolved once at top of `decide()` instead of per-rule. (#21)
- **Fixed-window counter**: Shared `FixedWindowCounter` eliminates duplicated increment logic. (#27)
- **Fail2Ban logic**: Shared increment-and-ban helper used by both pre-handler and post-handler (RequestContext) paths. (#29)

- **Fail2Ban/Allow2Ban threshold off-by-one**: Threshold comparison now uses `>=` consistently. (#26)
- **@pmFromFile path traversal**: Prevented directory traversal via `@pmFromFile` operator argument. (#20)
